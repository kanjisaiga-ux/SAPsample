# SAPsample 開発標準

## 1. 目的と適用範囲

本標準は、SAP S/4HANA 2023 FPS02（SAP_BASIS 758）へ abapGit で取り込む
クラシックABAPおよびABAP Cloudのサンプルに適用する。新規開発、修正、
レビューでは、本書と `ABAP_UNIT_TESTING.md` を必ず先に確認する。

優先順位は、動作の正しさ、保守性、セキュリティ、Clean Core、簡潔さの順とする。
サンプルであっても、本番コードへ流用され得る品質で作る。

## 2. リポジトリと開発フロー

- SAPで有効化したオブジェクトをabapGitでシリアライズし、
  `sap-learning-samples/src` 以下へ格納する。
- abapGitの主言語は日本語（`J`）とし、オブジェクト説明も原則日本語で作成する。
- abapGit管理ファイルを手作業で変更しない。例外は、生成形式を理解したうえで
  レビュー可能な小規模修正を行う場合だけとする。
- 1コミットは1つの論理変更とし、コード、ABAP Unit、文書を同時に更新する。
- ブランチ名は `feature/<topic>`、`fix/<topic>`、`docs/<topic>` を基本とする。
- コミットメッセージは `feat: ...`、`fix: ...`、`test: ...`、
  `docs: ...`、`refactor: ...` の形式を使う。
- 秘密情報、SAP接続先、ユーザー名、パスワード、実データ、社内固有情報を
  Gitへ保存しない。

## 3. 言語バージョンの選択

### 3.1 ABAP Cloud

新規の疎結合な拡張、RAP、公開APIだけで実装できる機能ではABAP Cloudを第一候補とする。

- オブジェクトのABAP言語バージョンは `ABAP for Cloud Development` とする。
- SAPがリリースしたAPI、型、CDSエンティティだけを利用する。
- 永続化アクセスは公開CDSまたは自作の公開済みエンティティを優先する。
- RAPではbehavior definition、service definition、service bindingの責務を分ける。
- released APIで実現できない場合は、クラシックABAPへの切替理由を文書化する。

### 3.2 クラシックABAP

既存拡張ポイント、未公開API、クラシックDynpro、既存資産との密結合が必要な場合に使う。

- SAP_BASIS 758で利用可能な構文を上限とする。
- obsoleteな構文や暗黙動作へ依存せず、可能な範囲でモダンABAPを使う。
- unreleased APIや標準テーブル直接参照は、代替手段、影響、理由を
  サンプルREADMEまたはADRへ記録する。
- 新規ロジックをユーザーEXIT、BAdI、レポートイベントへ直接詰め込まず、
  テスト可能なクラスへ委譲する。

## 4. 命名と構造

- サンプル固有オブジェクトの接頭辞は原則 `ZSA_` とする。
- クラス `ZCL_SA_<目的>`、インターフェース `ZIF_SA_<目的>`、
  例外クラス `ZCX_SA_<目的>`、テーブル `ZSA_<目的>` を基本とする。
- ローカルテストクラスは `LTC_<対象>`、テストダブルは `LTD_<役割>` とする。
- メソッド名は動詞から始め、意図が分かる名前にする。
- 1クラス1責務を基本とし、I/O、業務ルール、永続化を分離する。
- 公開APIにはABAP Docを付け、契約、例外、副作用を記述する。

## 5. コーディング規約

- キーワードは大文字、識別子は小文字を基本とし、ADT Formatterの結果を優先する。
- インデントは空白2文字、1行は原則120文字以内とする。
- インライン宣言、コンストラクタ式、文字列テンプレートを可読性が上がる場合に使う。
- `SELECT *` を避け、必要項目、絞込条件、キーを明示する。
- ループ内SELECTを避け、結合、CDS、まとめ読みを検討する。
- 内部テーブルは用途に合うキーと種別を明示する。
- `sy-subrc` を必要箇所で直ちに評価し、失敗を握りつぶさない。
- 予測可能な業務エラーは型付き例外またはRAPメッセージとして表現する。
- 空の `CATCH`、汎用的な `CATCH cx_root`、無条件の `COMMIT WORK` を禁止する。
- 時刻、ユーザー、UUID、外部通信、DBアクセスは依存性注入できる境界へ隔離する。
- 更新処理では権限、ロック、LUW、再実行時の整合性を明示的に設計する。

## 6. CDS・RAP

- CDS view entityを優先し、安定した意味を持つ名前と注釈だけを公開する。
- 認可が必要な業務データではDCLを設計し、`#NOT_REQUIRED` を安易に使わない。
- UI注釈をデータモデルへ過剰に集中させず、metadata extensionを検討する。
- RAPのvalidationは入力整合性、determinationは導出、actionは明示的操作に限定する。
- managed/unmanagedの選択理由をサンプルREADMEへ記録する。
- EML結果の `FAILED`、`REPORTED`、`MAPPED` を必要に応じて必ず処理する。

## 7. セキュリティと運用

- 外部入力は型、長さ、許容値、権限を境界で検証する。
- 動的SQL、動的呼出し、ファイル／HTTPアクセスには許可リストを使う。
- 個人情報や認証情報をログ、例外テキスト、テストデータへ出力しない。
- 更新系サンプルは対象データと副作用をREADMEへ明記し、既定では安全側に倒す。
- 性能に影響するコードは想定件数を定め、SQL MonitorやSATで検証可能にする。

## 8. Definition of Done

変更は次をすべて満たしたとき完了とする。

1. SAP_BASIS 758で構文チェックと有効化が成功する。
2. ABAP Cloud対象は言語バージョンチェックとreleased APIチェックに合格する。
3. 新規・変更ロジックにABAP Unitがあり、全テストが成功する。
4. ATCの対象バリアントでエラーがない。警告には理由がある。
5. `scripts/Test-Repository.ps1` が成功する。
6. 秘密情報や社内固有データを含まない。
7. サンプルの目的、前提、実行方法、期待結果を文書化している。

## 9. 例外管理

標準から外れる場合は、対象、理由、代替案、リスク、解消条件を
対象サンプルのREADMEまたは `docs/adr` のADRへ記録する。
「サンプルだから」は例外理由にしない。
